# Отчет по Спринту 6: Инструменты клиента и Валидация сертификатов

## Что было реализовано

В рамках 6-го спринта мы реализовали полноценную клиентскую часть и движок проверки цепочек сертификатов в соответствии с требованиями RFC 5280. Кроме того, была добавлена автоматическая проверка статуса отзыва сертификатов с умным переключением (fallback) между OCSP и CRL.

### 1. Валидация цепочек сертификатов
* Создан отдельный модуль `micropki/validation.py`, который реализует:
  * Проверку криптографических подписей по всей цепочке до доверенного корневого сертификата.
  * Проверку сроков действия сертификатов (с поддержкой кастомного времени валидации).
  * Обработку расширений: Basic Constraints (проверка флага CA), Path Length Constraint и Key Usage.
* В случае ошибки возвращается понятный и структурированный результат.

### 2. Проверка статуса отзыва (OCSP и CRL fallback)
* Реализован модуль `micropki/revocation_check.py` со следующей логикой:
  * Парсинг расширений AIA (для OCSP) и CDP (для CRL) из проверяемого сертификата.
  * Приоритетный запрос реального времени по протоколу OCSP.
  * В случае сбоя сервера OCSP или его отсутствия выполняется fallback на скачивание и проверку CRL списка (CDP/HTTP).

### 3. Обработка CSR запросов на стороне Центра Сертификации (Сервера)
* В `micropki/ca.py` доработана функция выпуска сертификатов `issue_end_entity`, чтобы она умела принимать CSR:
  * Извлечение Subject и Subject Alternative Names (SAN) напрямую из CSR запроса.
  * Проверка валидности подписи CSR и минимального размера ключа.
* В `micropki/repo.py` реализован HTTP-шлюз `POST /request-cert`, который позволяет отправлять CSR запросы и получать итоговые сертификаты без прямого доступа к файловой системе УЦ.

### 4. Инструменты клиента (CLI Client)
* Создан новый модуль `micropki/client.py` и его интеграция с главной `CLI` через новую группу подкоманд `micropki client`.
* Добавлены новые команды:
  * `gen-csr` — генерация ключевой пары и создание запроса на сертификат.
  * `request-cert` — отправка CSR на HTTP сервер репозитория для выпуска.
  * `validate` — полная или базовая валидация полученного сертификата.
  * `check-status` — проверка отзыва конкретного сертификата вручную.

## Статус
Все задачи 6-го спринта успешно реализованы и верифицированы. Сквозной тест (создание CSR, выпуск через API, успешная валидация, отзыв, отклонение валидации) выполняется корректно. Работы по Спринту 6 официально завершены.
