# Отчёт по Спринту 4 — MicroPKI

**Цель спринта:** реализовать полный цикл отзыва сертификатов и распространения CRL (RFC 5280): отзыв в БД, генерация и подпись CRL, выдача по HTTP, тесты и проверка OpenSSL (при наличии).

---

## 1) Что реализовано

### Новые модули (STR‑13)
- `micropki/revocation.py` — нормализация причин отзыва (10 кодов RFC 5280), обновление статуса в БД, подтверждение отзыва в CLI (`--force`).
- `micropki/crl.py` — построение CRL v2 через `CertificateRevocationListBuilder` / `RevokedCertificateBuilder`, расширения **Authority Key Identifier**, **CRL Number**, на уровне записи — **CRL Reason** (при наличии причины в БД), подпись алгоритмом, согласованным с ключом ЦС.

### Структура каталогов (STR‑15)
Под `--out-dir` создаётся каталог `crl/` (при инициализации Root/Intermediate и при выпуске leaf). Файлы по соглашению: `crl/root.crl.pem`, `crl/intermediate.crl.pem` (или путь из `--out-file` / `--ca` как путь к PEM).

### CLI (CLI‑18 — CLI‑20)
- `micropki ca revoke <serial>` — опции `--reason`, `--crl` (опциональный путь или авто по издателю), `--ca-pass-file` **обязателен**, если указан `--crl`; `--force`; `--next-update` для срока следующего обновления CRL при регенерации после отзыва; `--db-path`, `--out-dir`.
- `micropki ca gen-crl` — `--ca root|intermediate|<путь-к-сертификату>`, `--ca-key` (если `--ca` — файл), `--ca-pass-file`, `--next-update` (дней), `--out-file`, `--out-dir`, `--db-path`.
- `micropki ca check-revoked <serial>` — статус из БД; опционально `--crl <path>` для проверки наличия serial в PEM CRL.

Команды `ca issue-cert` и `ca issue-intermediate` не менялись по контракту (CLI‑21).

### База данных (DB‑5 — DB‑7)
- Таблица `certificates`: поля `status`, `revocation_reason`, `revocation_date` заполняются при отзыве.
- Новая таблица `crl_metadata` (`ca_subject`, `crl_number`, `last_generated`, `next_update`, `crl_path`) с уникальным `ca_subject`.
- Миграция: при `PRAGMA user_version < 2` таблица создаётся, версия поднимается до **2** (совместимость со старыми БД спринта 3).

### Репозиторий (REPO‑9 — REPO‑12)
- `GET /crl` — по умолчанию CRL промежуточного ЦС; `?ca=root` / `?ca=intermediate`.
- Дополнительно: `GET /crl/intermediate.crl`, `GET /crl/root.crl` (REPO‑10).
- `Content-Type: application/pkix-crl`; при отсутствии файла — **404**.
- Заголовки: `Last-Modified`, `Cache-Control: max-age=...` (из `nextUpdate` CRL, минимум 60 с), `ETag` (SHA‑256 содержимого).
- Логирование в том же формате, что и прочие HTTP‑запросы.

### Логирование (LOG‑9, LOG‑10)
- Отзыв: успех (serial, reason, время), предупреждение при повторном отзыве, ошибка при отсутствии serial.
- CRL: старт генерации, итог (количество записей, thisUpdate/nextUpdate, номер CRL, путь), ошибки при сбоях.
- LOG‑11 (JSON): оставлен существующий текстовый аудит; при необходимости JSON можно добавить в спринте 7.

### Тесты (TEST‑21 — TEST‑26)
- `tests/test_sprint4_crl.py`: жизненный цикл отзыва, отсутствующий serial, повторный отзыв, инкремент CRL Number, выдача CRL по HTTP и сравнение с файлом, `ca check-revoked`, нормализация reason.
- `tests/test_sprint3_repository.py` обновлён: перед сервером генерируется intermediate CRL; `GET /crl` ожидает **200**.
- `openssl crl -CAfile ... -noout` — тест с `pytest.mark.skipif`, если `openssl` не в PATH.

## 3) Команды для самостоятельной проверки (после подготовки PKI)

Предполагается уже созданы Root, Intermediate и хотя бы один leaf (как в спринтах 2–3), файлы паролей и `pki\micropki.db`.

**Важно для PowerShell:** не копируйте плейсхолдеры в угловых скобках (`<SERIAL>`) — символ `<` в PowerShell зарезервирован. Возьмите реальный `serial_hex` из JSON (`list-certs`) или используйте `$s = "69BEE1AC4AC1B70D"; micropki ca revoke $s ...`.

1. **Список сертификатов и serial leaf:**  
   `micropki ca list-certs --db-path .\pki\micropki.db --format json`

2. **Отзыв (пример serial — подставьте свой):**  
   `micropki ca revoke 69BEE1AC4AC1B70D --reason keyCompromise --force --db-path .\pki\micropki.db --out-dir .\pki`

3. **Проверка в БД:**  
   `micropki ca check-revoked 69BEE1AC4AC1B70D --db-path .\pki\micropki.db`

4. **Генерация CRL Intermediate:**  
   `micropki ca gen-crl --ca intermediate --out-dir .\pki --db-path .\pki\micropki.db --ca-pass-file secrets\inter.pass`

5. **Проверка CRL и повторная генерация (номер CRL должен вырасти):**  
   `micropki ca gen-crl --ca intermediate --out-dir .\pki --db-path .\pki\micropki.db --ca-pass-file secrets\inter.pass`  
   `openssl crl -in pki\crl\intermediate.crl.pem -inform PEM -text -noout` (если `openssl` в PATH; иначе см. README — однострочник на Python с `cryptography`)

6. **Проверка подписи CRL:**  
   `openssl crl -in pki\crl\intermediate.crl.pem -inform PEM -CAfile pki\certs\intermediate.cert.pem -noout`

7. **HTTP:** поднять `micropki repo serve ... --pki-dir .\pki` и выполнить `curl.exe -i http://127.0.0.1:8080/crl` — тело должно совпадать с локальным `pki\crl\intermediate.crl.pem`.

8. **Негативные сценарии:** несуществующий serial → код выхода ≠ 0; повторный отзыв → предупреждение и код 0.

9. **Автотесты:**  
   `pytest tests/test_sprint4_crl.py -v`  
   `pytest tests/ -v`

## 4) Ограничения / не входит в спринт

- Delta CRL, indirect CRL — не требуются (CRL‑8).
- Полная демонстрация `openssl s_client -crl_check` против реального TLS‑сервера — рекомендована в ТЗ как необязательная (TEST‑27).
