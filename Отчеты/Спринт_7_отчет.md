# Спринт 7 — Отчёт: Усиление безопасности PKI

## Цель спринта

Усиление MicroPKI против операционных угроз путём реализации комплексного аудит-логирования с криптографической целостностью, средств безопасности (ограничение запросов, симуляция Certificate Transparency, обнаружение компрометации ключей) и политик (размеры ключей, сроки действия, ограничения SAN).

## Реализованные компоненты

### 1. Система аудита (`audit.py`) — AUD-1..AUD-4, LOG-17..LOG-19

- **Формат**: NDJSON (newline-delimited JSON), файл `./pki/audit/audit.log`
- **Цепочка хешей**: SHA-256, каждая запись ссылается на хеш предыдущей
- Первая запись: `prev_hash = "0" * 64`
- Хеш вычисляется по каноническому JSON (отсортированные ключи, без пробелов)
- **Файл цепочки**: `./pki/audit/chain.dat` — хранит хеши для быстрой верификации
- **Верификация**: `verify_log()` — полная проверка целостности, возвращает индекс первой повреждённой записи
- **Запросы**: `query_log()` — фильтрация по времени, уровню, операции, серийному номеру
- **Потокобезопасность**: `threading.Lock` для многопоточного доступа
- **Обязательные события AUDIT**: инициализация CA, выпуск сертификатов, отзыв, генерация CRL, компрометация, нарушения политик
- **Защита от утечек**: приватные ключи и пароли **никогда** не записываются в аудит-лог

### 2. Политики безопасности (`policy.py`) — POL-3..POL-7

| Политика | Функция | Описание |
|----------|---------|----------|
| POL-3 | `check_key_size()` | RSA: root ≥4096, inter ≥3072, EE ≥2048; ECC: root/inter P-384, EE P-256+ |
| POL-4 | `check_validity_period()` | Root ≤3650, inter ≤1825, EE ≤365 дней |
| POL-5 | `check_san_policy()` | Wildcard запрещён по умолчанию; типы SAN ограничены по шаблону |
| POL-6 | `check_algorithm()`, `check_csr_algorithm()` | SHA-1 запрещён |
| POL-7 | `check_path_length()` | Intermediate pathlen должен быть 0 |

Все проверки интегрированы в `ca.py` — при нарушении выпуск блокируется, создаётся запись AUDIT уровня failure.

### 3. Ограничение запросов (`ratelimit.py`) — CTL-1, CLI-34

- **Алгоритм**: Token Bucket per client IP
- **Потокобезопасность**: `threading.Lock`
- **Интеграция**: FastAPI middleware в `repo.py` и `ocsp_responder.py`
- **Флаги CLI**: `--rate-limit` (запросов/сек), `--rate-burst` (максимальный burst)
- **Ответ при превышении**: HTTP 429 Too Many Requests + заголовок `Retry-After`

### 4. Симуляция Certificate Transparency (`transparency.py`) — CTL-2

- **Файл**: `./pki/audit/ct.log`, append-only, права 0o644
- **Формат строки**: `timestamp | serial_hex | subject_dn | sha256_fingerprint | issuer_dn`
- **Автоматическая запись** при каждом выпуске сертификата (Root CA, Intermediate CA, End-entity)
- **Проверка включения**: `verify_inclusion(serial_hex, ct_log_path)`

### 5. Компрометация ключей (`compromise.py`) — CTL-3, CTL-4, CLI-33

- **Команда**: `micropki ca compromise --cert <path> --force`
- **Действия**:
  1. Вычисление SHA-256 хеша SPKI (публичного ключа)
  2. Запись в таблицу `compromised_keys`
  3. Отзыв сертификата с причиной `keyCompromise`
  4. Создание записи AUDIT высокой важности
  5. Опциональная генерация экстренного CRL
- **Блокировка**: `is_key_compromised()` проверяет хеш ключа перед выпуском
- Любой CSR с тем же публичным ключом будет **отклонён**

### 6. Расширение базы данных — DB-10, DB-12

Новая таблица `compromised_keys`:
```sql
CREATE TABLE IF NOT EXISTS compromised_keys (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    public_key_hash TEXT UNIQUE NOT NULL,
    certificate_serial TEXT NOT NULL,
    compromise_date TEXT NOT NULL,
    compromise_reason TEXT NOT NULL
);
```

Создаётся автоматически в `init_database()`.

### 7. CLI — CLI-31..CLI-35

| Подкоманда | Описание |
|------------|----------|
| `audit query` | Запрос аудит-лога с фильтрацией (--from, --to, --level, --operation, --serial, --format, --verify) |
| `audit verify` | Полная проверка целостности аудит-лога (--log-file-path, --chain-file) |
| `ca compromise` | Симуляция компрометации ключа (--cert, --reason, --force, --db-path) |
| `repo serve` | Добавлены флаги --rate-limit, --rate-burst |
| `ocsp serve` | Добавлены флаги --rate-limit, --rate-burst |
