# Отчёт по Спринту 3 — MicroPKI

**Цель спринта:** Реализовать управление жизненным циклом сертификатов и базовое хранилище сертификатов, включая базу данных, отслеживание уникальных серийных номеров и REST API для обслуживания сертификатов и точек распространения CRL.

## 2. CLI (CLI-12 … CLI-17)


| Требование | Статус | Реализация |
|------------|--------|------------|
| CLI-12 `db init` | Выполнено | Создает схему SQLite базы данных. Идемпотентен: если таблица уже существует, не выдает ошибку. Аргумент `--db-path`.
| CLI-13 `ca list-certs` | Выполнено | Отображает все выпущенные сертификаты в виде таблицы (serial, subject, expiration, status). Поддерживает фильтрацию по статусу (`--status`) и формат (`--format`, по умолчанию таблица). |
| CLI-14 `ca show-cert <serial>` | Выполнено | Извлекает один сертификат из базы данных и выводит его PEM-содержимое в stdout. |
| CLI-15 `repo serve` | Выполнено | Запускает HTTP-сервер репозитория на FastAPI. Аргументы: `--host`, `--port`, `--db-path`, `--cert-dir`. Сервер работает до прерывания (Ctrl+C) и логирует запросы. |
| CLI-16 `repo status` | На рассмотрении | Поддержка PID-файла не реализована. Проверка состояния возможна через проверку занятости порта. |
| CLI-17 Автоматическая вставка при выпуске | Выполнено | Команды `ca issue-cert` и `ca issue-intermediate` автоматически вставляют сертификат в базу данных после выпуска. |

## 3. Реализация PKI (Интеграция с БД и серийные номера)


| Требование | Статус | Реализация |
|------------|--------|------------|
| PKI-13 Уникальный генератор серийных номеров | Выполнено | Генератор создает 64-битные целые числа. Старшие 32 бита — Unix timestamp (секунды), младшие 32 бита — CSPRNG. Уникальность обеспечивается за счет временной компоненты и случайных данных. |
| PKI-14 Автоматическая вставка в БД при выпуске | Выполнено | После подписания сертификата и записи на диск, его данные (serial, subject, issuer, сроки, PEM, статус) вставляются в таблицу `certificates` базы данных. |
| PKI-15 Схема базы данных | Выполнено | Используется SQLite. Таблица `certificates`: `id`, `serial_number INTEGER UNIQUE`, `subject`, `issuer`, `not_before`, `not_after`, `cert_pem`, `status`, `revocation_reason`, `revocation_date`, `created_at`. Индексы по `serial_number` и `status`. |
| PKI-16 Миграции базы данных | На рассмотрении | Поддержка миграций не реализована. Инициализация базы осуществляется через `db init` и создание схемы с нуля. |
| PKI-17 Обработка ошибок | Выполнено | Если вставка в БД не удалась (например, дублирование серийного номера), выпуск сертификата отменяется. Файл сертификата не записывается. Операция является атомарной. |

## 5. Базовый репозиторий (HTTP-сервер)


| Требование | Статус | Реализация |
|------------|--------|------------|
| REPO-1 Реализация HTTP-сервера | Выполнено | Используется FastAPI. Сервер запускается командой `repo serve` и работает до остановки. |
| REPO-2 Эндпоинт: `GET /certificate/{serial}` | Выполнено | Принимает серийный номер в шестнадцатеричном формате (регистронезависимо). Ищет сертификат в базе данных. При нахождении возвращает PEM-сертификат. При отсутствии — 404. Для отозванного сертификата возвращается 410 Gone. |
| REPO-3 Эндпоинт: `GET /ca/{level}` | Выполнено | Поддерживает `level = root` и `level = intermediate`. Возвращает соответствующий CA-сертификат из `--cert-dir`. При отсутствии файла — 404. |
| REPO-4 Эндпоинт: `GET /crl` | Выполнено | Заглушка для Спринта 4. Возвращает 501 Not Implemented с сообщением "CRL generation not yet implemented". |
| REPO-5 Статическая файловая подсистема | На рассмотрении | Поддержка fallback-доступа к файлам сертификатов напрямую из файловой системы не реализована. |
| REPO-6 Логирование | Выполнено | Каждый HTTP-запрос логируется на уровне INFO с меткой времени, методом, путем, IP-адресом клиента и статусом ответа. |
| REPO-7 Заголовки CORS | Выполнено | Сервер включает `Access-Control-Allow-Origin: *` для тестирования из браузерных инструментов. |
| REPO-8 Об��аботка ошибок | Выполнено | Некорректные серийные номера, отсутствующие параметры или неподдерживаемые методы возвращают соответствующие HTTP-статусы (400, 405) с описательным сообщением. |

## 8. Итог

Большинство требований Спринта 3 выполнено. Реализована основа для управления жизненным циклом сертификатов с использованием SQLite базы данных, FastAPI HTTP-сервера и уникального генератора серийных номеров. Ключевые функции, такие как автоматическая вставка при выпуске, атомарность операций и REST API для получения сертификатов, работают корректно. Некоторые дополнительные и необязательные требования (миграции, PID-файл, JSON-логи, полный набор негативных тестов) остаются не реализованными, что соответствует приоритетам проекта. Система готова к переходу к Спринту 4 — отзыву сертификатов и генерации CRL.
