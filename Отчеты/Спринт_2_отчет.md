# Отчёт по Спринту 2 — MicroPKI

**Цель спринта:** Расширить PKI: создать Intermediate CA (промежуточный удостоверяющий центр) и реализовать шаблонный движок для выпуска серверных, клиентских и code-signing сертификатов с расширениями X.509v3 и поддержкой Subject Alternative Name (SAN).

| Требование | Статус | Реализация |
|------------|--------|------------|
| PKI-6 CSR Intermediate CA | Выполнено | `csr.generate_intermediate_csr()` — PKCS#10 с BasicConstraints CA=TRUE + pathLen |
| PKI-7 Root подписывает CSR | Выполнено | `csr.sign_intermediate_csr()` — v3, serial CSPRNG, Issuer=Root, BC CA=TRUE pathLen, KU keyCertSign+cRLSign, SKI/AKI, SHA-256/384 |
| PKI-8 Шаблоны server/client/code_signing | Выполнено | `templates.get_template_extensions()` — различные BC, KU, EKU для каждого шаблона |
| PKI-9 SAN-парсинг | Выполнено | `templates.parse_san()` — dns, ip, email, uri → GeneralName |
| PKI-10 Вывод в PEM | Выполнено | Intermediate → `intermediate.cert.pem`; end-entity → `<CN>.cert.pem` |
| PKI-11 Генерация ключа leaf | Выполнено | RSA-2048 или ECC P-256 (зависит от CA); plain PEM с 0o600 + warning |
| PKI-12 Подписание внешнего CSR | Выполнено | `--csr` → `load_csr_pem()`, проверка подписи, выпуск сертификата |

### Детали шаблонов (PKI-8)

| Шаблон | BasicConstraints | KeyUsage | ExtendedKeyUsage | SAN |
|--------|-----------------|----------|-------------------|-----|
| server | CA=FALSE, critical | digitalSignature + keyEncipherment(RSA), critical | serverAuth | Обязателен ≥1 dns/ip |
| client | CA=FALSE, critical | digitalSignature, critical | clientAuth | email (опционально) |
| code_signing | CA=FALSE, critical | digitalSignature, critical | codeSigning | Не обязателен; dns/uri |

## 7. Оптимизация кодовой базы

После реализации функциональности Sprint 2 был проведён полный аудит проекта. Выявлены и устранены следующие проблемы без изменения структуры спринтов.

### 7.1. Устранение дублирования кода

| Что дублировалось | Где | Решение |
|-------------------|-----|---------|
| Генерация серийного номера | `certificates.py` (inline, 4 строки) и `csr.py` (`_make_serial`, 3 строки) — разная логика | Единая функция `crypto_utils.make_serial()` |
| Выбор алгоритма подписи (RSA→SHA-256, ECC→SHA-384) | `certificates.py`, `csr.py`, `chain.py` — по 3-4 строки в каждом | Единая функция `crypto_utils.signing_algorithm()` |
| Генерация ключей (`if rsa ... else ecc`) | `ca.init_root_ca()`, `ca.issue_intermediate_ca()`, `ca.issue_end_entity()` + inline P-256 | Единая функция `crypto_utils.generate_key(type, size)` |
| Верификация подписи сертификата (RSA/ECC ветвление) | `chain.py` (`verify_signature`) и `ca.py` (`verify_certificate`) — по 8 строк | Единая функция `crypto_utils.verify_cert_signature()` |
| `KeyUsage(...)` с 9 параметрами | Копировался 5 раз: `certificates.py`, `csr.py`, `templates.py` (×3) | Хелпер `templates._leaf_key_usage()` + константа `_LEAF_BC` |
| Сохранение CA-артефактов (ключ + сертификат + permissions) | Идентичные блоки в `init_root_ca()` и `issue_intermediate_ca()` | Выделен `ca._save_ca_artifacts()` |
| Проверка overwrite (цикл по файлам) | Идентичный код в 2 функциях | Выделен `ca._check_overwrite()` |

### 7.2. Удаление deprecated API

- **`default_backend()`** убран из 8 мест (`crypto_utils.py`). В `cryptography` ≥ 38.x этот параметр не нужен и помечен как deprecated.

### 7.3. Исправление утечки handlers в логгере

- **Проблема:** каждый вызов `setup_logging()` удалял старые handlers без `h.close()` и создавал новые, даже если конфигурация не менялась. При множественных вызовах это приводило к накоплению незакрытых file descriptors.
- **Решение:** добавлено кеширование текущей конфигурации (`_current_log_file`), handlers закрываются через `h.close()`, повторная настройка пропускается если config не изменился.

### 7.4. Вынос inline-импортов

- **`ca.py`:** 6 inline `from . import ...` и `import re` внутри функций — вынесены на уровень модуля.
- **`crypto_utils.py`:** inline `from cryptography import x509` — вынесен на уровень модуля.

### 7.5. Добавление `from __future__ import annotations`

Добавлен во все модули `micropki/` для forward-compatible type hints. Позволяет использовать `str | None` вместо `Optional[str]` даже в Python 3.8+.

### 7.6. Data-driven SAN-парсинг

- **Было:** цепочка `if/elif/elif/elif` в `parse_san()` и `validate_san_for_template()`.
- **Стало:** словари `_SAN_PARSERS` и `_SAN_TYPE_MAP` — добавление нового типа SAN — одна строка.

### 7.7. Расширение поддержки ECC

- `generate_ecc_key()` теперь принимает и 256 (P-256), и 384 (P-384). Ранее при передаче 256 выбрасывалось исключение, хотя Sprint 2 требует P-256 для leaf-ключей.
